# Agent Boundaries

Rules that apply to every task. See [AGENTS.md](../../../AGENTS.md) for overview; [CONTEXT.md](./CONTEXT.md) for what to do before changing code.

## Always

- Read the relevant `README.md` before making changes
- Make **minimum changes only** — don't refactor unrelated code
- Run linting and tests after changes
- When you change behavior, update the relevant tests
- Include the MPL-2.0 file header on every new Swift file (enforced by SwiftLint `file_header` rule):
  ```swift
  // This Source Code Form is subject to the terms of the Mozilla Public
  // License, v. 2.0. If a copy of the MPL was not distributed with this
  // file, You can obtain one at http://mozilla.org/MPL/2.0/
  ```
- Follow SwiftLint rules defined in `.swiftlint.yml`
- Await user confirmation before pushing or making commits

## Ask First

Before doing any of these, ask the user:

- Adding new dependencies
- Modifying Firefox core files (anything outside `Ecosia/` and `Client/Ecosia/`)
- Schema or API changes
- Changes to `BrowserKit/` shared package
- Creating additional documentation, quickstarts, or ADRs

## Never

- Commit secrets, API keys, or credentials
- Remove failing tests without fixing the root cause
- Reassign `Analytics.shared` outside of tests (enforced by SwiftLint custom rule `reassign_analytics_instance`)
- Force unwrap (`!`) except in controlled scenarios (tests, IBOutlets)
- Edit `Client.xcodeproj` directly — it is generated by Tuist
- Create summary files or next-steps documents — provide summaries in chat instead
- Add Ecosia-specific strings to Firefox's `Shared/Strings.swift`
